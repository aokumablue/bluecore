# ADR-0001: コンポーネント起動検証は scratchpad 隔離＋承認ゲート方式とする

**日付**: 2026-07-12  **ステータス**: accepted

## コンテキスト

インストール済み bluecore プラグイン（0.9.17: agents 16 / commands 10 / skills 13 / hooks 29 エントリ / output-styles 1）の全コンポーネント動作検証を実施した。hooks は実起動しないと非ブロッキング契約・exit code・stderr の実態を確認できないが、実起動には副作用（DB 書き込み・ファイル生成・git 操作）が伴い、検証対象のプラグインファイルや開発リポジトリを汚染するリスクがある。また検証で発見した NG の修正はリポジトリ変更を伴うため、検証（報告）と修正（変更）の境界を明確にする必要があった。

## 決定

コンポーネント起動検証は、書き込み系を scratchpad フィクスチャに隔離して実起動し（hooks 35 ケースを launcher 経由で同期実行、プラグインファイルは一切未修正・報告のみ）、repo 変更を伴う後段フェーズ（NG 修正など）はユーザー承認ゲートで停止する方式とする。

## 検討した代替案

### 代替案 1: 実リポジトリ対象の完遂検証

- 長所: 修正まで一気通貫で完了し、再実行の手間がない。実環境そのままの挙動を確認できる
- 短所: 検証中の副作用が開発リポジトリ・プラグインキャッシュを汚染する。検証（事実の収集）と修正（意思決定）が混ざり、意図しない変更が承認なしに入り込む
- 却下理由: 検証フェーズの READ-ONLY 性が崩れ、レポートの信頼性（「未修正の 0.9.17 を検証した」という前提）が担保できないため

### 代替案 2: 静的精読のみ

- 長所: 副作用ゼロで安全。追加のフィクスチャ整備が不要
- 短所: hooks の非ブロッキング契約・exit code・タイムアウト・import 整合など実行時挙動を検証できず、トレースバックや実害 NG を見逃す
- 却下理由: 実起動レイヤ（hooks）の検証が目的の中心であり、静的精読だけでは検証の意味を成さないため

## 結果

### 肯定的

- hooks 35 ケースの実起動検証を副作用なしで完遂（exit code / stderr / タイムアウト整合まで実測）
- 検証レポートが「プラグイン未修正」を前提にでき、報告の信頼性が担保される
- repo 変更は必ずユーザー承認を経るため、検証と修正の責任境界が明確

### 否定的

- scratchpad フィクスチャの整備コストがかかる（入力 JSON・環境変数・venv 解決の再現）
- 承認ゲートで停止するため、修正完了までのリードタイムが延びる

### リスク

- scratchpad 環境と実環境の差異（パス・権限・DB 接続）により、実環境でのみ再現する不具合を見逃す可能性
- agents / commands / skills は定義精読による擬似起動に留まるため、実行時にのみ現れる問題は検出範囲外
